Current Setup

Before MCP

The current chart imports a single Lifecycle realm and configures browser login for the UI plus token validation for Lifecycle Core. It does not currently define a dedicated MCP public client or Dynamic Client Registration policy.

User browser Signs in through Lifecycle UI using Keycloak authorization code flow.
→
Keycloak realm: lifecycle Hosts roles, clients, IdP broker config, and user profile attributes.
→
Lifecycle app APIs Verify access tokens with issuer, JWKS, and existing audience rules.

Client lifecycle-ui

Confidential OIDC client. Uses a client secret and fixed UI redirect URLs.

  • standardFlowEnabled: true
  • Redirects to clients.lifecycleUi.url and /*
  • Audience mapper includes lifecycle-core

Client lifecycle-core

Public bearer-style client used as the existing API audience target.

  • standardFlowEnabled: false
  • directAccessGrantsEnabled: false
  • Included as audience on UI tokens

IdP GitHub

Linked GitHub identity provider stores GitHub data as Keycloak user attributes.

  • githubUsername user attribute
  • githubProfilePicture user attribute
  • UI client maps github_username into tokens

What We Are Adding

Disabled by default

The chart patch adds an optional MCP auth surface. Nothing changes unless custom values enable it. The new blocks are intentionally separate so operators can override clients, scopes, claims, and DCR policy behavior independently.

MCP client Codex, VS Code, Cursor, Claude Code, Inspector, or another agent harness.
→
Keycloak additions lifecycle-mcp client, lifecycle-mcp scope, and anonymous DCR policies.
→
Lifecycle MCP endpoint https://app.../mcp verifies tokens with MCP resource audience.

1 Public MCP client

Adds clients.lifecycleMcp, a public PKCE client for fallback OAuth when a client cannot complete DCR.

  • clientId: lifecycle-mcp
  • standardFlowEnabled: true
  • pkce.code.challenge.method: S256
  • Loopback redirect defaults for local CLI clients

2 MCP client scope

Adds clientScopes.lifecycleMcp, which emits the claims the Lifecycle MCP server expects.

  • aud includes the MCP resource URL
  • github_username maps from githubUsername
  • Extra protocol mappers can be supplied from custom values

3 DCR policies

Adds clientRegistrationPolicies.anonymous so URL-only clients can dynamically register.

  • Trusted hosts: 127.0.0.1, localhost
  • Registration web origins for MCP Inspector
  • Allowed client scopes and max client limit

4 Extension points

Keeps the template maintainable by exposing common override lists rather than hardcoding every future client detail.

  • clients.lifecycleMcp.attributes
  • clients.lifecycleMcp.protocolMappers
  • clientScopes.lifecycleMcp.protocolMappers
  • clientRegistrationPolicies.anonymous.extraPolicies

Auth Flows

MCP clients can either use Dynamic Client Registration or fall back to the preconfigured public client. Both flows end with a token that the Lifecycle MCP endpoint validates.

URL-first OAuth with Dynamic Client Registration

1
User adds URL

https://app.lifecycle.lfc.goodrx.com/mcp

2
MCP endpoint challenges

WWW-Authenticate points at protected-resource metadata.

3
Client discovers Keycloak

Reads authorization server metadata and registration endpoint.

4
DCR registers client

Keycloak policy checks trusted host, origins, and allowed scopes.

5
Token is validated

Lifecycle checks issuer, JWKS, resource audience, and user access.

Why DCR matters URL-only setup means the user does not manually copy a client id or bearer token. The MCP client must be able to register a safe OAuth client automatically, then launch browser login with PKCE.

Fallback using the preconfigured lifecycle-mcp client

MCP client Uses configured or discovered client_id=lifecycle-mcp.
→
Keycloak browser login PKCE with loopback redirect URI from redirectUris.
→
Lifecycle MCP server Accepts token only when aud includes the MCP resource URL.

Custom Values Shape

For the umbrella lifecycle chart, the MCP Keycloak values sit under the existing keycloak: subchart key. The chart defaults stay off; production turns them on explicitly.

Single-environment rollout posture Lifecycle currently has one production Keycloak environment. These values are additive and do not delete the existing realm, clients, users, roles, secrets, or IdP configuration. They still change the auth surface by enabling a new public client and optional DCR policies, so rollout should use rendered manifest review, helm diff, and Keycloak CR status checks rather than relying on a lower environment. 
keycloak:
  clients:
    lifecycleMcp:
      enabled: true
      resourceUrl: 'https://app.lifecycle.lfc.goodrx.com/mcp'

  clientScopes:
    lifecycleMcp:
      enabled: true
      audienceMapper:
        audience: 'https://app.lifecycle.lfc.goodrx.com/mcp'

  clientRegistrationPolicies:
    anonymous:
      enabled: true
      trustedHosts:
        hosts:
          - 127.0.0.1
          - localhost
      allowedRegistrationWebOrigins:
        origins:
          - 'http://localhost:6274'
          - 'http://127.0.0.1:6274'
Value Default Reason
clients.lifecycleMcp.enabled false Backward compatible. Existing Keycloak installs do not render the MCP client until a release enables it.
clients.lifecycleMcp.resourceUrl http://localhost:3000/mcp Feeds the MCP audience mapper unless a custom audience override is supplied.
clientScopes.lifecycleMcp.enabled false Can be enabled with the client, or independently if another client should attach the same scope.
clientRegistrationPolicies.anonymous.enabled false DCR is additive, but it has security impact, so it is opt-in and tightly configured by values.
extraPolicies, attributes, protocolMappers empty lists or objects Lets operators extend without editing the template for every future MCP client need.

Compatibility And Access Notes

Likely supported

These are standard Keycloak OIDC features and should work on the current runtime.

  • Public OAuth client with authorization code and PKCE
  • Client scopes
  • Audience protocol mapper
  • User attribute mapper for github_username

Needs environment proof

The risky part is the exact DCR policy representation accepted by Keycloak 26.4.7 realm import.

  • trusted-hosts policy provider id
  • registration-web-origins policy provider id
  • allowed-client-templates policy provider id
  • Policy config key names in imported realm components
Users without github_username They can authenticate and can use MCP capabilities that only require an authenticated Lifecycle identity. They should not be able to use build-owner capabilities when ownership depends on GitHub PR author matching github_username. For those owner-scoped tools, resources, and prompts, the correct result is an empty owned-build list or 403, not broader access.
MCP capability type Requires github_username? Expected behavior
General authenticated account, docs, schema, or user-profile capabilities No Allow any valid authenticated Lifecycle user, subject to ordinary role checks.
Admin operations No, if the token has the admin role Admin role is sufficient. GitHub ownership is not needed for admin-scoped access.
Build-owner tools and resources, such as v0.1 build lookup, pods, webhooks, metadata, and redeploy Yes, unless admin Non-admin users without github_username fail closed for owner-scoped build access.
Prompts that call owner-scoped tools or resources Indirectly yes The prompt can be listed, but execution should only succeed when the underlying tools and resources pass access checks.

Validation checklist

  1. Render the exact production values with helm template and review the diff for additive-only realm changes.
  2. Run helm diff against the production release before applying.
  3. Confirm the realm import creates the lifecycle-mcp client and lifecycle-mcp client scope.
  4. Confirm a token for MCP includes aud: https://app.../mcp and github_username when the user has the attribute.
  5. Confirm a non-admin user without github_username can authenticate but cannot access owner-scoped build capabilities.
  6. Confirm anonymous DCR accepts loopback MCP clients and rejects untrusted hosts.
  7. Run MCP Inspector and Codex login against the MCP endpoint.
